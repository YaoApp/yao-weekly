# Yao 0.9.1 正式推出

经过很长一段时间筹备之后，Yao 0.9.1 正式发布！

我们不敢说 Yao 有多么伟大，但事实是，我们确实创造了一个新生生物，在开源世界，还没有 Yao 这种**逻辑驱动的前后端一体化的低代码开发框架**，我们希望通过 Yao，给应用开发提供另一种可能性，这种可能性可以让企业和开发者在这个快速变化的时代通过大幅提高生产力而占得一份先机，同时也让万物互联时代，早点到来，让我们每个人都享受到数字化的便利。

Yao 包含以下几个核心特性，以此带给开发者全新，且难以想象的高效开发体验：

- **无代码**：使用 JSON 创建数据库模型、编写接口、描述界面，真正意义上的无代码编程。

- **万物互联**：支持在 ARM 设备上运行，可用于物联网、边缘计算、工业互联网。

- **数据可视化**：简单易用的 BI 组件和功能，赋予商业更多的可能性。

- **处理器**：内置一系列可以通过数据流、Javascirpt 和 GRPC 插件进行功能扩展的处理器。

- **数据流**：通过数据流，开发者可以对数据进行任意形式的处理，且支持 Javascript。

- **插件**：基于 GRPC 的插件机制，支持使用 nodejs、python 等语言进行功能拓展。

- **管理后台**：通过编写 JSON 描述界面布局，快速制作各种 CRM、ERP 等企业内部系统。

- **实时查询**：内置媲美 ES 的查询引擎，且通过 DSL 实现 JSON 查询，简单且易用。

- **极致性能**：底层使用 Go 语言，性能远超 Java、PHP，分布式部署，一行命令搞定。

更多特性请看[文档](/doc)。

## Yao 是什么？

Yao 是一个支持快速创建 Web 服务和管理后台的开源低代码应用引擎。

Yao 提供开箱即用的 CLI，创建数据库、编写接口、描述界面，执行 `yao start`，即可搭建一个后台应用。与低代码搭建页面不同，Yao 提供完备的后端逻辑表达能力，以支撑多变的业务场景。

## 为什么会做 Yao？

互联网二十年，我们从蛮荒走到现在，从刀耕火种一般的封闭式开发，到现在开源驱动的合作共赢模式；从 C、C++到 PHP、.Net、JAVA，再到如今的 Nodejs、Go、Rust；从 ThinkPHP、Symphony、Spring 到 Laravel、Spring Cloud、Gin、Beego；从 Jsp、Mustache、Ejs、Handlebars 到 Emberjs、Angularjs、Vue、React、Nextjs，再到如今的 Alpine、Svelte；从 Gulp、Grunt，到 Rollup、Webpack、Babel，再到 Esbuild、Swc、Vite。

Dev 领域也在发生着变革，从 Docker 开始，全面上云。游戏开发领域也发生了翻天覆地的变化，Unity 和 Unreal Engine 的诞生意味着游戏开发进入了它的“工业化时代”。

应用领域的“工业化时代”什么时候到来？现在的开发模式能变得更好、更高效吗，现在的开发模式是互联网工业的最终形态吗？

我们见过太多项目，大部分时间都花在沟通、“对齐”上，留给人们做产品设计和应用开发的时间因此被挤压，这种依靠人们通过不停的开会“对齐”的开发模式是 OK 的吗？

有没有一种开发模式，能够把开发者从重复且无效的沟通中解脱出来，真正专注于业务本身？

我们曾经也为上面的这些问题而苦恼，之后我们开始尝试代码生成方案，在很长的一段时间里，代码生成方案给我们节省了不少时间，但这还不够，我们需要更加规范的，通用的，且更加强大的代码生成方案，你可以把它当作一种低代码方案，但在我们眼里，不会纠结于低代码的概念，而是最终能够把很多机械的、重复的工作能够丢给机器去做，解放人。

而且，我们发现，针对一些企业数字化还有物联网场景的需求，他们需要的只是一个能够方便管理数据和进行数据分析的后台，而这里面的很多逻辑还有界面都是通用的，一些熟练的工程师通过复制粘贴就能够很快实现目的，但因为人的因素，也会出现一些疏漏，也会有 bug，但这种问题其实完全可以避免。

于是在代码生成器的基础上，我们让前端界面也通过动态组件+配置的方式生成，于此同时，由于是后端解析配置，我们直接通过服务器来实现一些配置的转化，比如把描述的接口转化成数据然后在渲染时直接发送给组件进行渲染，通过给 Table 组件设计很多可选属性来支持各种各样的场景，而你，只需要在 JSON 中声明对应的属性。

后来我们加入了`Process`[处理器](/doc/b.基础特性/d.使用处理器)，`Dataflow`[数据流](/doc/b.基础特性/e.编写数据流)，来实现更加复杂的业务场景，为了实现一些复杂的查询，设计了[`Guo Query DSL`](/doc/c.高级特性/c.复杂数据查询)，将 JSON 映射为 Sql，且提供很多的快捷操作。在各种各样的业务场景中，我们提炼出了 5 中类型页面，超过 30 个动态组件，你可以称之为“云组件”，以及将近 80 个内置处理器，我们还内置了 V8 这个 JS 引擎，你可以通过 JS 来实现自定义的`Process`和`Dataflow`，至此，Yao 应运而生。

我们没有刻意去想要做一个只给别人用的工具，那对我们没有任何价值，我们最初的目的纯粹是自用，直到我们意识到，我们可能创造了一个新的物种，而它，可能会颠覆现在的应用开发模式，就像[福特发明汽车流水线](https://baike.baidu.com/item/%E7%A6%8F%E7%89%B9%E7%94%9F%E4%BA%A7%E6%B5%81%E6%B0%B4%E7%BA%BF/4122269)，流水线带来了生产力的大幅提高，人类加速进入工业化时代，而 Yao 的使命是，让万物互联的时代，早点到来。

道家讲：“上善若水”，而我经常喜欢说一句话：“水到渠成”，即顺应事物的自然发展规律，不烦躁，不焦虑，不揠苗助长，不掩耳盗铃，顺着本心，一步一个脚印，走出一条路，无论这条路有没有人走过，走出自己的风采，走出自己的价值。

于是在经过很长时间的实践和打磨之后，我们选择将 Yao 开源，你可以认为它是一个命令行工具、一个 Serverless 服务器、一个比 Nginx 更加强大的网关、一个代码生成器、一个数据库管理系统，通过你的“加工”，它也可以是一个 CMS、一个 CRM、一个 ERP、一个后台管理系统，甚至，基于 Yao，你可以很轻松的打造一个低代码开发平台。

我们不生产想法，我们只为你实现想法提供锤子、榔头，你也可以把 Yao 当作车床，如果你的企业规模较大，你可以基于 Yao 打造一个应用开发流水线——而这也是我们内部的常规用途。

## 为什么会开源？

### 敢为天下先

坦白讲，在这个时代，想要被大家看见，已经是一件很难的事情了，所以会存在很多劣币驱逐良币的情况，而在开源世界，则会好很多，所以我们放弃去打造平台，转而去做开源，市面上的低代码平台又很多，开源的有几个？

你说你比别人的好，你说你是新物种，那开源，大家比一比，碰一碰，切磋切磋，好的会变得更好，这种模式最终让 Yao 变得更好。

### 开发者关系

在知乎上，我注意到“低代码”相关话题的回答，80%的回答趋于负面，60%的回答是一种高高在上的心态在谈论着一件似乎与自己毫不相关的事情，而 20%的正面或者中性回答也大都在宣传自己的平台有多好。

“低代码”已被污名化。

除了上述一些原因，更多的是很多程序员发自内心的抵触，这种抵触导致低代码的消费人群从来都不是什么开发者，而是企业，也导致低代码大部分以平台这种形式存在。

程序员不需要低代码吗？用 Nodejs + React 手写一套博客后台至少得 1 个星期，而且拓展功能的时候还是要写代码，数据库迁移，自动化部署这些都需要你自己去做，如果使用 Yao，一个 Blog 后台，30 分钟搞定。

程序员需要低代码，万物互联的时代更加需要低代码，如果你觉得低代码这个词不好听，你可以把 Yao 当作一个代码生成器，只不过是一个功能有点强大的代码生成器。

科技的进步不因人的意志而停滞，历史的车轮滚滚向前，你我能做的，就是在一个时代落幕，下一个时代到来之前，尽力发出自己的光芒，照亮前路，也照亮他人。

## 未来的路

就像 Angularjs 那样，Yao 在推出之后可能面临各种各样的模仿者，这些模仿者可能会比 Yao 在某些方面做的更好，所以我们不会就此停住脚步。

小步快跑，这是我从一家我待过的企业学习到的一种精神和做事情的方式，在未来的日子里，我们可以保证的是，Yao 将一直保持更新，只要我们一直在做生意。我们也会开放一部分功能给到社区当中有贡献有实力的开发者，我们不会鼓吹开源精神，我们做事情的目的很简单：

Make things better。
